# Group 18 - WebUI Module Video Scripts

## WebLayout - Alex Elkins

This is the web layout class, it interfaces with the DOM to manage the layout of other web objects. The web layout creates a div to hold the other elements and includes plenty of options to customize the appearance and layout of the div. For layout options, you can choose between free, horizontal and vertical flexbox, or grid positioning, as well as justify, align, and space the content in grid or flex layouts, and set margin and padding. When elements are added they can also override their individual alignment if they wish. There are also options for styling like background color, borders, box-shadow, and opacity. You can also toggle the visibility of the layouts to show or hide content when necessary.

## WebTextbox - Lance Motowski

WebTextbox represents a logical text element in our Web UI layer and wraps a single HTML div element through emscripten::val. The class implements IDomElement, which means it participates in the layout system but does not manage its own positioning. Core API methods include SetText, AppendText, Clear, and GetText. These modify the internal std::string and then synchronize it to the DOM. Formatting is handled through setters such as SetFontFamily, SetFontSize, SetBold, SetItalic, SetColor, SetAlignment, SetWrap, and SetMaxWidth. These update internal state and are applied to the DOM through a centralized ApplyStyles() function. The RectPx struct represents a bounding box in pixel space. GetBoundingBoxPx() queries the browser using getBoundingClientRect() to determine the rendered size and position. Mounting is handled through mountToLayout. The textbox does not append itself to the DOM body directly, WebLayout owns structure. The constructor generates a unique id like webtextbox-1, creates a div via document.createElement, assigns the id, and sets display to inline-block so bounding box measurements are stable. Importantly, it does not append to the document body. This enforces separation of responsibilities between element behavior and layout. ApplyText() updates textContent on the DOM element. ApplyStyles() maps stored C++ style state to DOM CSS properties. This centralizes styling logic so setters do not directly manipulate the DOM. Bounding box measurement calls getBoundingClientRect() through Emscripten and returns width and height in pixels. This enables layout-aware sizing and integration with other UI components. Move semantics transfer ownership of the underlying DOM handle safely. The moved-from object nulls its emscripten::val so we never double-own the same DOM element. The test suite validates both logical state and DOM-backed behavior. State tests verify text modification, visibility toggling, id uniqueness, and move logic. DOM-level tests validate bounding box behavior, mounting behavior, and style application. In the demo, textboxes are created, styled, and mounted via layout, demonstrating typical usage for HUD elements, logs, and overlays.

## WebButton - Tung Hoang

WebButton is a C++ class in Group 18's Web Interface module that manages an HTML button element through Emscripten. From C++, you can control the label, size, colors, enabled/disabled state, visibility, and attach a click callback without writing any JavaScript. The class implements the IDomElement interface, a shared contract across our module that WebImage, WebTextbox, and WebButton all use, so WebLayout can manage them uniformly. Copy is deleted since each instance owns a unique DOM element, but move semantics are supported. Each button gets a unique ID using a static counter so WebLayout can find elements through the DOM. WebButton does not handle its own positioning - WebLayout handles that entirely. For error handling, asserts catch programmer mistakes like negative sizes or null callbacks, while clicking a disabled button silently does nothing. The demo shows three buttons in a vertical WebLayout. The first is clickable with a console callback, the second is disabled, and the third is hidden. Testing includes 30 Catch2 test cases with 91 assertions, all passing, covering construction, callbacks, state management, styling, move semantics, and IDomElement compliance.

## WebImage - Feichi Cai

WebImage binds a C++ object to a browser image element. It implements both the IDomElement and ICanvasElement interfaces, so it works with the layout system and also resolves a canvas path. In the constructor, an image element is created with a unique ID, src, and alt text, then appended to document.body. AttachListeners connects browser image events back to C++. The destructor removes the DOM node from its parent to avoid leftover elements. Copy is disabled and move semantics are supported, so each instance has unique ownership of one DOM element. For content updates, SetSource and SetAltText update both the model and the DOM. SetSource also resets the load and error state. For sizing, SetSize forces stretch, while Resize can preserve aspect ratio using object-fit contain. Visibility and appearance are straightforward - SetOpacity, Show, and Hide map directly to CSS style changes. The key bridge is in the JS/WASM layer, where JavaScript registers load and error listeners and calls back into exported C functions. Those exported functions dispatch to HandleLoad and HandleError, which update internal state and run user callbacks. On failure, the image swaps to a blank texture or a color placeholder. WebImage exposes parent element, ID, and alignment. WebLayout controls positioning, and WebImage controls image state and styling. All tests pass with around 110 assertions across 35 test cases.

## WebCanvas - Pinhe Du

WebCanvas sits between WebLayout and the other UI elements in our architecture. It manages an HTML5 canvas element and provides a C++ API for drawing dynamic content. The class implements IDomElement for integration with WebLayout, and also acts as a canvas manager that owns ICanvasElement objects and dispatches draw calls in renderFrame. Elements are sorted by z-index using stable sort, so elements with the same z-index keep their insertion order. Invisible elements are skipped during rendering. WebCanvas is move-only with deleted copy operations, same pattern as the other classes in our module. The constructor takes an optional ID string, and if none is provided, defaults to "web-canvas." For the JavaScript side, the JS code is separated into a standalone JS library file rather than inlined in C++. This follows Emscripten conventions and keeps things cleaner as the JS code grows. When compiling, the JS file is passed with the --js-library flag. For testing, there are 8 Catch2 test cases with 70 assertions, all passing. The tests cover move-only type traits, IDomElement compliance, DOM lifecycle safety, element ownership and cleanup, render frame behavior, null element handling, visibility filtering, and z-index sorting. The demo shows a canvas drawing a background color, XY axes, a circle, and a rotating triangle, demonstrating how straightforward it is to draw on a web canvas from C++.
